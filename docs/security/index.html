🛡️ 脅威モデル

前提と信頼境界

  • •
    分散実行環境: 実行ノードは信頼できない前提で設計。計算レシートによる事後検証を基本とします。
  • •
    ネットワーク攻撃者: 通信路上の攻撃者は暗号化により防御。中間者攻撃はNoise Protocol Framework (IK pattern)で防止。
  • •
    悪意のあるProvider: 不正な結果返却は署名検証で検出。リソース枯渇攻撃はレート制限とステーキングで抑制。

🔒 データ取扱いポリシー

転送時の保護

すべての通信はQUIC/TLS 1.3またはNoise Protocol Frameworkで暗号化されます。

// Noise IK pattern (既知の相手との通信)
-> e, es, s, ss
<- e, ee, se
// 前方秘匿性を確保しつつ、相手の公開鍵を事前検証

実行時の取扱い

  • ⚠️
    基本モード: 推論実行時、Providerノード上でプロンプトは平文処理されます。機密データは信頼できるProviderを選択するか、追加保護を適用してください。
  • 🔐
    TEEモード (開発中): Intel SGX/AMD SEV等のTrusted Execution Environmentでの隔離実行。
  • 🔀
    分割実行 (計画中): Secure Multi-party Computationによる分散処理。

🐛 脆弱性開示プログラム (VDP)

報告方法

  1. 脆弱性の詳細を記載したレポートを作成
  2. 可能であればPoCコードを添付（悪用しない範囲で）
  3. 下記PGP鍵で暗号化して送信: security@quiver.network
  4. 72時間以内に受領確認、7日以内に初期評価をお返しします

対象範囲

  • ✓
    Provider/Gateway/Aggregatorの各コンポーネント
  • ✓
    P2P通信プロトコルと暗号実装
  • ✓
    計算レシートの署名・検証ロジック
  • ✓
    スマートコントラクト（決済層）

除外事項

  • 既知の依存関係の脆弱性（ただし実証可能な影響がある場合は除く）
  • ソーシャルエンジニアリング
  • 物理的なアクセスが必要な攻撃
  • 理論的な攻撃で実証困難なもの

🔨 再現可能ビルド

すべての公式リリースは再現可能ビルドです。以下の手順で検証できます：

# ソースコードの取得
git clone https://github.com/yukihamada/quiver
cd quiver
git checkout v1.2.0

# Dockerによる再現ビルド
docker build -t quiver-verify -f Dockerfile.reproducible .
docker run --rm quiver-verify sha256sum /app/bin/*

# 公式リリースのハッシュと比較
curl -L https://registry.quiver.network/v1.2.0/checksums.txt
